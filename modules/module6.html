Модуль 6: Тестування API — Запити до "Серця" Програми

Ми переходимо від тестування видимого інтерфейсу (UI) до тестування "мозку" додатку — його API. Якщо UI — це зал ресторану, то API — це кухня, куди ми будемо робити замовлення напряму. Для цього ми використовуємо потужну бібліотеку Python — `requests`.

Основи API: CRUD та HTTP-методи

В основі більшості API лежить концепція CRUD (Create, Read, Update, Delete), яка реалізується через HTTP-методи:

  • Create (Створити): Метод `POST`. Відправляємо дані на сервер для створення нового об'єкта.
  • Read (Прочитати): Метод `GET`. Запитуємо існуючі дані з сервера.
  • Update (Оновити): Метод `PUT`. Повністю замінюємо існуючий об'єкт новими даними.
  • Delete (Видалити): Метод `DELETE`. Видаляємо об'єкт з сервера.

Аутентифікація: API-ключі

Більшість API захищені. Найпростіший спосіб аутентифікації — це API-ключ. Це унікальний "пропуск", який ми додаємо до наших запитів, щоб сервер знав, хто ми, і чи є у нас дозвіл на отримання даних.


# Приклад передачі ключа як параметра запиту
api_key = "ВАШ_СЕКРЕТНИЙ_КЛЮЧ"
params = {
  "q": "Kyiv",
  "appid": api_key 
}
response = requests.get("https://api.openweathermap.org/...", params=params)

Коди статусу HTTP

Кожна відповідь від сервера містить код статусу, який показує результат операції:

  • 2xx (Успіх): `200 OK` (Все добре), `201 Created` (Створено новий ресурс).
  • 4xx (Помилка клієнта): `400 Bad Request` (Неправильний запит), `401 Unauthorized` (Проблема з ключем), `404 Not Found` (Не знайдено).
  • 5xx (Помилка сервера): `500 Internal Server Error` (Щось зламалось на сервері).
