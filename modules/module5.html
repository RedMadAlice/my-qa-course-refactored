Модуль 5: Git та GitHub — Твоя машина часу

Професійна розробка неможлива без системи контролю версій. Уяви, що ти пишеш складний тест, і раптом все ламається, а ти не пам'ятаєш, що саме змінила. Або ти випадково видалила важливий файл. Git — це твоя страхова сітка і машина часу, яка дозволяє робити сміливі експерименти, знаючи, що завжди можна повернутись у безпечну точку.

Як "думає" Git: Три дерева

Щоб по-справжньому зрозуміти Git, потрібно знати про три "простори", між якими переміщуються твої зміни. Це найважливіша концепція.

  1. Робоча Директорія (Working Directory): Це звичайна папка з твоїми файлами на комп'ютері. Це твій "робочий стіл", де ти можеш створювати, редагувати, видаляти файли. На цьому етапі Git просто спостерігає за змінами.
  2. Індекс / Staging Area (Область підготовки): Це "проміжний буфер". Уяви, що ти готуєш посилку. Ти не кидаєш в коробку все підряд. Ти обираєш, що саме покласти. Staging Area — це і є ця "посилка". Команда git add <файл> бере файл з твого робочого столу і кладе його в цю посилку.
  3. Репозиторій (Repository, папка `.git`): Це твій "архів" або "фотоальбом". Коли посилка готова, ти її "запечатуєш" і ставиш дату. Команда git commit бере все, що лежить у Staging Area, робить з цього знімок (коміт) і зберігає його в архіві назавжди.

Твій щоденний цикл: 1. Змінив файли на столі -> 2. Поклав потрібні зміни в посилку (`git add`) -> 3. Запечатав посилку і поклав в архів (`git commit`).

Гілки (Branches): Працюємо паралельно, не заважаючи іншим

Уяви, що основна гілка main — це офіційна, стабільна версія твого проєкту. Золоте правило: ніколи не працюй напряму в `main`!

Коли ти починаєш нову задачу (наприклад, пишеш тести для сторінки логіну), ти створюєш власну, окрему гілку. Це ніби ти робиш копію проєкту, в якій можеш експериментувати, нічого не зламавши в основній версії. 


# Створити нову гілку і одразу перейти в неї
git checkout -b feature/login-tests

Тепер ти в своїй ізольованій "пісочниці". Ти можеш робити коміти, і вони будуть лише в твоїй гілці. Коли задача готова, ти створюєш **Pull Request** на GitHub — це прохання "прийняти мої зміни з гілки `feature/login-tests` в основну гілку `main`".

Професійний робочий процес (Workflow)

Ось як виглядає твій типовий робочий день:

  1. Синхронізація: Перш ніж почати, завантаж останні зміни з `main`.
    git checkout main
    git pull
  2. Створення гілки: Створи нову гілку для своєї задачі.
    git checkout -b feature/some-new-task
  3. Робота: Пиши код, змінюй файли.
  4. Збереження прогресу: Періодично роби коміти.
    git add .
    git commit -m "Feat: add basic login test"
  5. Відправка в хмару: Коли готовий показати роботу, відправ свою гілку на GitHub.
    git push -u origin feature/some-new-task
  6. Pull Request: Зайди на GitHub і створи Pull Request для об'єднання твоїх змін з `main`.

Що таке "коміт" (commit) в Git?

Резервна копія всього комп'ютера.
Знімок стану файлів у певний момент часу, "точка збереження".
Коментар у коді.

Коміт — це ключове поняття в Git. Це не просто зміна, а повний знімок (snapshot) проєкту в конкретний момент часу з унікальним ID та описом.

Яка команда покаже поточний стан репозиторію: які файли змінено, які готові до коміту?

git commit
git add
git status

`git status` — це твоя "приладова панель". Цю команду використовують найчастіше, щоб зрозуміти, що відбувається в репозиторії перед тим, як робити будь-які дії.

У чому головна відмінність між Git та GitHub?

Це одне й те саме.
Git — це програма на комп'ютері, а GitHub — це веб-сайт для хостингу репозиторіїв.
Git — для програмістів, а GitHub — для дизайнерів.

Це фундаментальна відмінність. Git — це інструмент (як молоток). GitHub — це місце, де ти зберігаєш проєкти, зроблені цим інструментом (майстерня).

Що робить команда `git add .`?

Створює новий коміт з усіма змінами.
Додає всі змінені та нові файли в поточній папці до "staging area" (підготовки до коміту).
Відправляє всі зміни на GitHub.

Це другий крок у "святій трійці" Git. Команда `git add` готує зміни до збереження. Вона не робить сам "сейв" (коміт), а лише каже: "Я хочу, щоб ось ці зміни були включені в наступне збереження". Крапка (`.`) означає "всі файли в поточній директорії".

Яка команда використовується для відправки локальних комітів на віддалений репозиторій (напр., GitHub)?

git commit
git pull
git push

Команда `git push` (дослівно "штовхати") бере всі коміти, які ви зробили локально, і відправляє їх на віддалений сервер, такий як GitHub, роблячи ваші зміни доступними для інших або просто створюючи резервну копію в хмарі.
